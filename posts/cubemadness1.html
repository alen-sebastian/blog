HTB CubeMadness1 — Cheating My Way to the Flag

A GamePwn challenge where the goal is to collect 20 cubes in a game that only has 6. Spoiler: you're not meant to find the rest.

I'll be honest — when I first saw the GamePwn category on HTB I almost skipped it. I do web stuff mostly. Game hacking felt like a completely different world. But CubeMadness1 is rated Very Easy and I had twenty minutes to kill, so I downloaded the zip and opened it up. What followed was actually one of the more satisfying "oh it's that simple?" moments I've had in a while.

What even is this challenge

You extract the zip and get a folder full of DLLs, a couple of executables, and some Unity game assets. Running the main executable opens up a small 3D game — you're a little character on a platform, and there are cubes scattered around. The score counter in the top-left reads 0 / 20.

I ran around the map collecting everything I could find. Got to 6. Map was empty. 6 / 20 sitting there mocking me. There are literally no more cubes anywhere on the map.

So the game is lying to you — or rather, it's telling you the win condition is 20 but only giving you the tools to reach 6. Which means you're not supposed to find the other 14. You're supposed to make the game think you already have them.

My first instinct (the wrong one)

My first thought was to reverse the game binary. The game's built with Unity, and when I poked at the GameAssembly.dll, I could tell pretty quickly it was compiled with il2cpp. That's Unity's way of converting C# game logic to native C++ — developers use it specifically to make the game harder to reverse. Tools like Il2CppDumper can help, but for a Very Easy challenge this felt like overkill.

I closed dnSpy and opened Cheat Engine instead.

What Cheat Engine actually does

If you've never used it — Cheat Engine is a memory scanner for Windows processes. It attaches to a running program and lets you search through its memory for specific values, then modify them on the fly. Game hackers have used it forever to do things like set their health to 9999 or give themselves unlimited ammo. For us, it's a legitimate RE tool for runtime analysis.

The logic we're working with: somewhere in the game's memory, there's an integer that holds the current cube count. We need to find it and change it to 20.

Attaching to the process and initial scan

With the game running, I opened Cheat Engine and hit the process selector (the flashing monitor icon in the top left). Selected the CubeMadness1 process. Now I'm looking at the game's memory in real time.

My current cube count is 0. So I set the scan type to Exact Value, value type to 4 Bytes (standard integer — most game scores are stored this way), and searched for 0.

The result: roughly 40 million addresses all holding the value 0. Obviously. Everything in the process that's zero right now shows up. This is expected — the first scan is just to establish a baseline.

Narrowing it down

This is where it gets interesting. I went back into the game and collected exactly one cube. Counter now reads 1 / 20. I switched the scan type to Increased value by... and entered 1, then clicked Next Scan. CE threw out every address that didn't increase by exactly 1. Down to around 37,000.

Still too many. Collected a second cube. Next Scan again with value increased by 1. Down to a few thousand. Collected the third, fourth — kept scanning. After collecting all 6 available cubes and filtering each time, I was left with just 3 addresses.

All three held the same current value (6) and had changed consistently with my cube count through every scan. That's the address — or rather, three copies of it, which Unity sometimes keeps in sync across different memory regions. 

# The 3 narrowed-down addresses after collecting all 6 cubes
# All three holding value: 6
# Example addresses (yours will differ each run):
0x206FEA32A40
0x206FEA32A44
0x206FEA32A48

The actual "exploit"

I selected all three addresses in the CE results panel, hit the red arrow to send them to the address list at the bottom, then double-clicked each value field and changed them to 20.

Switched back to the game window.

The counter instantly jumped to 20 / 20 and the flag appeared on screen. 

HTB{CU83_M4DN355_UNM4DD3N3D}

Why this works

Games track state in memory like any other program — they just do it fast and in real time. The cube count is just an integer. There's no server validating it, no checksum, no anti-cheat layer. We're running this locally, so we own the process. Cheat Engine just gives us a systematic way to find which integer it is among the millions sitting in memory.

The multi-pass scanning technique (scan for 0 → update → scan for 1 → update...) is the core of how Cheat Engine narrows things down. Every time the value you're tracking changes by a known amount, you filter out everything that didn't match. After a handful of iterations you're left with the exact addresses you care about.

Would I try CubeMadness2?

Yeah, already did. The second challenge in the series adds a layer of encryption on top of the memory values — so the raw integer you see in CE isn't the actual cube count, it's an encoded form of it. Same concept, significantly more annoying to pull off. I'll write that one up soon.

This one though — genuinely a good intro to GamePwn. If you've never touched Cheat Engine before, this challenge teaches you the fundamentals of memory scanning in about twenty minutes, which is a solid return on time investment.

If you're new to game hacking, the Cheat Engine built-in tutorial (it ships with one) is actually worth going through before trying CTF challenges. It covers exactly this kind of value scanning. Questions? Find me on Twitter or email.
